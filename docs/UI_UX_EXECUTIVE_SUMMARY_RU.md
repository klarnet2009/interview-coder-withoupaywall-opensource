# Executive Summary: UI/UX аудит и план

Дата: 2026-02-07  
Проект: `interview-coder-withoupaywall-opensource`

## 1) Короткий вывод

Продукт решает сильную задачу: ускорение ответа кандидата в live-интервью через screenshot/audio -> AI pipeline.  
Текущее UX-состояние рабочее, но под стрессовым использованием страдает предсказуемость:
- пользователь не видит этапы ожидания;
- часть onboarding-проверок симулирована;
- тексты и фактическое поведение местами расходятся.

Главный фокус редизайна: не "сделать красивее", а сделать интерфейс операционно надежным в условиях давления времени.

## 2) Топ-10 критичных проблем

1. Нет визуализации `processing-status` при наличии backend-событий.
2. Конфликт hotkeys в UI-копирайте и реальных shortcuts.
3. "Тесты" в wizard создают ложную уверенность (симуляции).
4. Несоответствие security-обещаний фактическому хранению API key.
5. Непоследовательная семантика `Log out`.
6. Перегруженный верхний action-row с мелкими target controls.
7. Слабая иерархия важности между critical и utility действиями.
8. Чрезмерно мелкая типографика в ключевых местах.
9. Недостаточная error-recovery логика (toast без next action).
10. Смешение актуального UX с legacy-остатками в кодовой базе.

## 3) Что даст наибольший эффект при минимальных усилиях

## P0 (немедленно)

1. Подключить отображение `processing-status` в renderer.
2. Привести hotkeys к одному источнику правды.
3. Исправить security copy на технически точный.
4. Убрать/маркировать fake-тесты в wizard.
5. Унифицировать поведение `Log out`.

Ожидаемый эффект:
- резкий рост доверия к интерфейсу;
- снижение "silent confusion";
- меньше ошибок управления в live-сценарии.

## 4) Новая UX-рамка (концепт)

Базовая модель:

```text
Idle -> Capture Ready -> Processing -> Responding -> Follow-up
Idle -> Live Listening -> Transcribing -> Generating -> Responding
Any state -> Error -> Recovery -> Previous stable state
```

Ключевой экран: Session HUD
- крупные primary actions;
- явная статусная дорожка;
- поток частичных результатов;
- контекстные recovery-действия.

## 5) Дорожная карта

1. Спринт 1 (P0, 2-3 дня): правда в статусах, hotkeys, security copy, logout.
2. Спринт 2 (P0/P1, 3-4 дня): latency masking + переработка action-row.
3. Спринт 3 (P1, 3 дня): честный onboarding с реальными проверками.
4. Спринт 4 (P1, 2-3 дня): упрощение IA и cleanup legacy поверхностей.
5. Спринт 5-6 (P1/P2, 4 дня): accessibility, QA, smoke сценарии.

## 6) Критерии успеха

1. Time to first useful answer снижается.
2. Пользователь всегда видит, на каком этапе pipeline.
3. First-run setup проходит без "ложно успешных" тестов.
4. Ошибки hotkey-навигации и misclick уменьшаются.
5. Recovery после ошибок становится предсказуемым.
